Data Analyst Cheat Sheet

Quick answer cues for common analyst interviews. Keep answers short, structured, and tied to business decisions.

Answer frameworks Core questions

Tell Me About Yourself

  • Present: current role, training, or focus.
  • Past: relevant projects, tools, or domain exposure.
  • Strength: how you solve business questions with data.
  • Future: why this role is the right next step.

Behavioural Answer Frame

  • Situation: what was happening.
  • Task: what you needed to solve.
  • Action: what you did and why.
  • Result: metric, learning, or business effect.

Project Story Frame

  • Problem -> Data -> Analysis -> Insight -> Outcome
  • Problem: what decision needed support.
  • Data: sources, size, quality issues.
  • Outcome: action, impact, next step.

02

SQL Interview Cheat Sheet

Keep SQL explanations simple. Show logic, not only syntax. Think in tables, row grain, filters, joins, and aggregation order.

How to Think Through an SQL Question

  • Define final output: rows, columns, grain.
  • Identify source tables and join keys.
  • Filter early where possible.
  • Aggregate only after the row set is correct.
  • Check duplicates and NULL impact.
  • Explain edge cases aloud.

Common SQL Patterns Analysts Use

  • Daily KPI trends
  • Top categories by metric
  • Funnel step counts
  • Retention cohorts
  • Latest status per user
  • Percent of total

SQL Cheat Memory Wall

  • SELECT = choose columns
  • WHERE = filter rows
  • GROUP BY = summarize
  • HAVING = filter after summarize
  • INNER JOIN = matched rows
  • LEFT JOIN = keep left rows
  • CTE = named step
  • WINDOW = calculate without collapsing rows

SELECT / WHERE

Concept: Pull columns and filter rows.

Memory cue: Start narrow. Only select what you need.

SELECT user_id, country
FROM users
WHERE signup_date >= '2026-01-01';

Mistake to avoid: Filtering after aggregation when `WHERE` should be used first.

GROUP BY / ORDER BY

Concept: Summarize rows, then sort output.

Memory cue: Group first, order last.

SELECT country, COUNT(*) AS users
FROM users
GROUP BY country
ORDER BY users DESC;

Mistake to avoid: Selecting a non-aggregated column that is not in `GROUP BY`.

JOINS

Concept: Combine tables using matching keys.

Memory cue: INNER = matched records. LEFT = keep all left rows.

SELECT o.order_id, c.customer_name
FROM orders o
LEFT JOIN customers c
  ON o.customer_id = c.customer_id;

Mistake to avoid: Joining on the wrong grain and creating duplicates.

CASE WHEN

Concept: Create logic-based labels or buckets.

Memory cue: CASE is SQL if-else.

CASE
  WHEN score >= 80 THEN 'high'
  WHEN score >= 50 THEN 'mid'
  ELSE 'low'
END AS score_band

Mistake to avoid: Overlapping logic that puts rows in the wrong bucket.

AGGREGATE FUNCTIONS

Concept: Reduce many rows into summary values.

Memory cue: COUNT, SUM, AVG, MIN, MAX.

SELECT COUNT(*) AS orders,
       SUM(revenue) AS total_revenue
FROM orders;

Mistake to avoid: Forgetting `COUNT(column)` ignores NULLs.

HAVING

Concept: Filter grouped results.

Memory cue: HAVING works after `GROUP BY`.

SELECT country, COUNT(*) AS users
FROM users
GROUP BY country
HAVING COUNT(*) > 100;

Mistake to avoid: Using `HAVING` for row filters that belong in `WHERE`.

SUBQUERIES

Concept: Use one query inside another.

Memory cue: Good for filtering or comparing to a derived set.

SELECT user_id
FROM users
WHERE user_id IN (
  SELECT user_id FROM orders
);

Mistake to avoid: Writing a subquery when a join or CTE would be clearer.

CTEs

Concept: Break a query into named steps.

Memory cue: CTE = readable staging area.

WITH daily_orders AS (
  SELECT order_date, COUNT(*) AS orders
  FROM orders
  GROUP BY order_date
)
SELECT * FROM daily_orders;

Mistake to avoid: Many unclear CTEs without a clean flow.

WINDOW FUNCTIONS

Concept: Calculate across related rows without collapsing them.

Memory cue: Window keeps detail rows.

SELECT user_id,
       order_date,
       SUM(revenue) OVER (
         PARTITION BY user_id
         ORDER BY order_date
       ) AS running_revenue
FROM orders;

Mistake to avoid: Confusing grouped output with windowed output.

RANK / DENSE_RANK / ROW_NUMBER

Concept: Number rows within a partition.

Memory cue: `ROW_NUMBER` never ties. `RANK` skips. `DENSE_RANK` does not.

ROW_NUMBER() OVER (
  PARTITION BY country
  ORDER BY revenue DESC
)

Mistake to avoid: Forgetting tie behavior in top-N questions.

DATE FUNCTIONS

Concept: Group or compare time periods.

Memory cue: Analysts often bucket by day, week, month.

SELECT DATE_TRUNC('month', order_date) AS month,
       SUM(revenue)
FROM orders
GROUP BY DATE_TRUNC('month', order_date);

Mistake to avoid: Comparing incomplete periods to full periods.

NULL Handling

Concept: NULL means missing or unknown, not zero.

Memory cue: Use `IS NULL`, not `= NULL`.

SELECT COALESCE(discount, 0) AS discount
FROM orders
WHERE cancelled_at IS NULL;

Mistake to avoid: Treating NULL as a normal value.

DISTINCT / UNION vs UNION ALL

Concept: `DISTINCT` removes duplicate rows. `UNION` removes duplicates across queries. `UNION ALL` keeps all rows.

Memory cue: Duplicate removal costs work.

SELECT DISTINCT country FROM users;
SELECT city FROM a
UNION ALL
SELECT city FROM b;

Mistake to avoid: Using `UNION` when duplicates are acceptable and performance matters.

Primary Key / Foreign Key Basics

Concept: Primary key uniquely identifies a row. Foreign key links to another table.

Memory cue: Key logic prevents bad joins.

Mistake to avoid: Assuming uniqueness without checking it.

Data Cleaning with SQL

Concept: Standardize text, handle blanks, cast types, remove duplicates.

Memory cue: Clean before trusting counts.

SELECT TRIM(LOWER(email)) AS clean_email
FROM users;

Mistake to avoid: Mixing blank strings and NULLs without normalization.

KPI Calculations in SQL

Concept: Turn user actions into ratios and trends.

Memory cue: Numerator, denominator, period, grain.

SELECT
  COUNT(DISTINCT purchaser_id) * 1.0
  / COUNT(DISTINCT visitor_id) AS conversion_rate
FROM funnel_daily;

Mistake to avoid: Mixing time periods or mismatched denominators.

Product Metrics Cheat Sheet

Product analytics is about behavior, value, and change over time. Tie every metric back to the user journey and product objective.

Core Metric Frame

  • Objective -> Metric -> Trend -> Insight -> Recommendation
  • Observation -> Insight -> Recommendation

North Star Metric

  • One metric that best reflects delivered user value.
  • Should connect user success and business growth.
  • Needs guardrails to avoid unhealthy optimization.

Guardrail Metrics

  • Metrics that protect quality while chasing growth.
  • Examples: churn, complaint rate, latency, cancellation rate.
  • They answer: are we growing the right way?

Product Objective

  • Question: What user or business outcome are we trying to improve?
  • Examples: Increase activation, reduce churn, improve course completion.
  • Why it matters: Metrics without an objective create noise.

User Journey

  • Awareness -> signup -> activation -> engagement -> retention -> referral or revenue.
  • Ask where value starts and where friction appears.
  • A drop suggests a journey problem, not just a metric problem.

Funnel Metrics

  • Sign-up rate: visitors who register.
  • Activation rate: new users who reach first value.
  • Drop-off rate: users lost at a step.
  • Increase suggests: better clarity or less friction.
  • Drop suggests: poor onboarding, bugs, weak intent, or tracking issues.

Engagement Metrics

  • DAU / WAU / MAU: active users by period.
  • Stickiness: DAU / MAU.
  • Session frequency: how often users return.
  • Feature adoption: share of users who use a feature.
  • Rise may mean: stronger value or better relevance.
  • Drop may mean: weak usefulness, fatigue, or usability issues.

Retention Metrics

  • Retention rate: users who come back after a period.
  • Churn: users who leave or stop being active.
  • Cohort retention: compare user groups by signup date.
  • Strong retention means: product keeps delivering value.
  • Weak retention suggests: poor activation, low fit, or unmet expectations.

Conversion Metrics

  • Conversion rate: users who complete the desired action.
  • Completion rate: users who finish a process or content flow.
  • Cart-to-purchase or lead-to-paid: indicates execution quality.
  • Drop may suggest: friction, trust issues, price, or broken flow.

Revenue / Business Metrics

  • Revenue per user: average revenue from each user.
  • CAC: customer acquisition cost.
  • LTV: lifetime value.
  • Why it matters: Growth is stronger when value exceeds acquisition cost.
  • Watch for: growth that looks good but is too expensive.

Learner / Platform Metrics

  • Completion rate: how many learners finish content.
  • Time to first value: how quickly users get useful output.
  • Content return rate: whether users come back to learn more.
  • Marketplace thinking: balance supply, demand, fill rate, and repeat behavior.

Experimentation Basics

  • Start with a clear hypothesis and primary metric.
  • Use guardrails so wins do not hide damage elsewhere.
  • Check sample size, duration, and segment effects.
  • A result can be statistically valid but still not strategically useful.

Product Insight Formula

  • Observation: activation fell 8% week over week.
  • Insight: most loss came from mobile users at the profile step.
  • Recommendation: simplify the step and validate with an experiment.

Satisfaction Metrics

  • NPS: willingness to recommend.
  • CSAT: satisfaction after a touchpoint.
  • Why it matters: behavior tells what users did; sentiment helps explain why.

Trend Reading

  • A metric rise is not always good. Check quality and guardrails.
  • A metric drop is not always bad. It could reflect a cleaner definition or lower spam.
  • Always ask: compared to what, for whom, and since when?

Case Study Framework

In a case study, structure matters as much as the analysis. Show how you think, what you checked, what you found, and what should happen next.

Main Framework

  • Problem -> Data -> Analysis -> Insight -> Recommendation
  • Objective -> KPI -> Breakdown -> Findings -> Action
  • Question -> Metric -> Slice -> Compare -> Explain -> Recommend
  • Observation -> So What -> Now What

If Given a Dataset, Start Here

  • Clarify the business question.
  • Check row count, columns, keys, date range.
  • Confirm grain and definitions.
  • Look for missing data and duplicates.
  • Only then start deeper analysis.

Strong Presentation Phrases

  • The data suggests...
  • One possible driver is...
  • This matters because...
  • A practical next step would be...
  • I would validate this by...

Understand the Problem

  • What decision needs support?
  • Who is the stakeholder?
  • What does success mean?
  • What constraints matter?

Choose the Right KPIs

  • Pick one primary KPI tied to the objective.
  • Add supporting metrics for drivers.
  • Add guardrail metrics to avoid one-sided conclusions.

Check Data Quality

  • Look for missing values, duplicates, and broken joins.
  • Check date coverage and tracking changes.
  • Call out quality risks early.

Segment the Data

  • Slice by user type, region, channel, device, or product line.
  • Segmentation often reveals the real story.
  • Do not rely on one overall average.

Find Trends and Anomalies

  • Compare periods over time.
  • Look for spikes, drops, seasonality, and breaks in pattern.
  • Ask what changed operationally or in product.

Compare Groups or Periods

  • Before vs after
  • Control vs treatment
  • New vs returning users
  • High vs low-performing segments

Check the User Journey

  • Map the funnel from entry to outcome.
  • Find the biggest drop-off step.
  • Ask whether the problem is acquisition, onboarding, or retention.

Draw Business Implications

  • Translate numbers into business meaning.
  • Say who is affected and why it matters.
  • Keep recommendations realistic and scoped.

Present Clearly to Stakeholders

  • Lead with the answer, not the process.
  • Use only the visuals that support the story.
  • Close with decisions and next steps.

5-Slide Presentation Structure

  1. Business problem and objective
  2. Data and method
  3. Key findings
  4. Insight and business meaning
  5. Recommendation and next step

7-Step Checklist Before Presenting

  1. Recheck metric definitions.
  2. Confirm date ranges.
  3. Validate totals against source logic.
  4. Check segment comparisons.
  5. Remove clutter from slides.
  6. Prepare one recommendation.
  7. Prepare one validation step.

Power BI Dashboard Thinking

Interviewers are usually testing thinking, not buttons. Show that you can design for audience, business questions, and action.

Dashboard Story Formula

  • What happened -> Why it matters -> What to do next
  • Good dashboards support this story at a glance.

What Should Every Dashboard Answer?

  • What is happening now?
  • How is it changing over time?
  • Where is the biggest issue or opportunity?
  • Which segment needs attention?
  • What action should follow?

How to Explain a Dashboard to a Manager

  • Start with the objective.
  • Point to headline KPIs.
  • Show the main trend.
  • Highlight one key breakdown.
  • End with decision options.

Before You Build

  • Who will use it?
  • What decisions will it support?
  • What are the non-negotiable KPIs?
  • How fresh does the data need to be?

Dashboard Layout Logic

  • Top: KPI cards for the headline view.
  • Middle: trend charts for movement over time.
  • Bottom: breakdowns, tables, or root cause views.
  • Side or top: filters only if they help decisions.

Audience-First Design

  • Executives want summary and movement.
  • Managers want breakdowns and operational drivers.
  • Analysts may need detail tables and drill-downs.

Dashboard Building Blocks

  • KPI cards: headline numbers
  • Trends: change over time
  • Breakdowns: compare categories
  • Filters / slicers: focus the view
  • Insights: annotate what matters
  • Actions: point to next steps

Best Visual Choices

  • Line chart = trend over time
  • Bar chart = compare categories
  • Stacked bar = composition
  • Card = headline KPI
  • Table = detailed lookup
  • Scatter = relationship between variables
  • Avoid pie charts unless categories are few and obvious.

Filters / Slicers Thinking

  • Use only filters people will actually need.
  • Do not overload with every possible field.
  • Default states should still tell a useful story.

Storytelling with Dashboards

  • Show the KPI.
  • Show the trend.
  • Show the segment driving the change.
  • Add a short insight note.

Dashboard Mistakes to Avoid

  • Too many visuals on one screen
  • Weak hierarchy and tiny text
  • Unclear metric definitions
  • Too many colors competing for attention
  • Decorative charts without a decision purpose

Power BI Interview Talking Points

  • Start from audience and business questions.
  • Explain KPI selection and layout choices.
  • Mention data model quality and refresh reliability.
  • Show that visuals were chosen for interpretation speed.

Common Dashboard Interview Questions

How do you design for executives?How do you choose KPI cards?What chart would you use for category comparison?How do you avoid clutter?How do you explain a dashboard to a manager?What would you improve in a bad dashboard?

Example Dashboard Sections

  • Product: activation, retention, feature adoption, funnel drop-off
  • Sales: revenue, conversion, average order value, top regions
  • Operations: turnaround time, backlog, SLA, issue volume
  • Learning platform: enrollments, completion, active learners, lesson drop-off
